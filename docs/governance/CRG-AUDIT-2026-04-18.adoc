== Scope and Standard

Evaluates the `neurophone` repository against *Component Readiness Grades
v2.0 (STRICT)*, 2026-03-30 revision. Neurophone was flagged in this
session's Bucket-B sweep as a "plausibly defensible" self-declared C —
the repo's own numbers are internally consistent and do not obviously
contradict the C claim, so it merits the full evidence audit.

- Standard: `standards/component-readiness-grades/COMPONENT-READINESS-GRADES.md`
- Criteria boilerplate: `rsr-template-repo/docs/governance/CRG-CRITERIA.adoc`
- Audit template: `rsr-template-repo/docs/governance/CRG-AUDIT-TEMPLATE.adoc`
- Self-declared grade (prior art):
  * `.machine_readable/6a2/STATE.a2ml` `[metadata] crg-grade = "C"`,
    declared 2026-04-04 with commit note "CRG C grade achieved".
  * `[project-context] completion-percentage = 65`.
- Audit date: 2026-04-18
- Auditor: Jonathan D.A. Jewell

Schema note: neurophone uses the *flat* `crg-grade = "C"` under
`[metadata]` (unlike gossamer's sectioned `[crg] grade = …`). This is the
form HYP-S005 already matches — no rule drift here.

The audit grades the repo *as-is today*, not aspirationally.

=== CATASTROPHIC FINDING: committed unresolved merge conflicts

`git status` on the working tree is clean (`nothing to commit`) and HEAD
is at `75bdc66` (2026-04-17). Yet *20 files are committed to main with
unresolved `<<<<<<< HEAD … ======= … >>>>>>>` merge-conflict markers*.
These were introduced by `5fe78cf Merge branch 'main' of
github.com:hyperpolymath/neurophone` and carried forward silently.

Files with committed conflict markers (verified by grep for
`^<<<<<<< HEAD` / `^>>>>>>> `):

[cols="2,3"]
|===
| Class | Files (20 total)

| Top-level docs
| `README.adoc`, `MAINTAINERS.adoc`, `TESTING-REPORT.adoc`, `TESTING-REPORT.scm`

| GitHub plumbing
| `.github/dependabot.yml`, `.github/FUNDING.yml`

| *Every RSR workflow that was touched in the merge*
| `.github/workflows/codeql.yml`, `guix-nix-policy.yml`, `instant-sync.yml`, `language-policy.yml`, `mirror.yml`, `quality.yml`, `rsr-antipattern.yml`, `rust-ci.yml`, `scorecard.yml`, `scorecard-enforcer.yml`, `secret-scanner.yml`, `security-policy.yml`, `wellknown-enforcement.yml`, `workflow-linter.yml`
|===

Sample (`.github/workflows/rust-ci.yml` lines 1-5):

[source,yaml]
----
<<<<<<< HEAD
# SPDX-License-Identifier: PMPL-1.0-or-later
=======
# SPDX-License-Identifier: MPL-2.0-or-later
>>>>>>> 0a372d5 (chore: sync from parent repo automation)
name: Rust CI
----

A YAML file cannot parse with `<<<<<<< HEAD` as its first token. This
means every affected workflow (including `rust-ci.yml`, `codeql.yml`,
`scorecard.yml`, `quality.yml`, `security-policy.yml`) is *syntactically
invalid* and cannot have run successfully since the merge. The README
badge "image:https://img.shields.io/…[License: PMPL-1.0]" is in a
conflict block — GitHub's adoc renderer will display the conflict
markers verbatim to any visitor.

This single finding invalidates *three* CRG-C hard requirements
simultaneously:

. **RSR compliance** (D-floor): workflows exist but don't parse, so they
  don't execute. RSR structural requirements are nominally met but
  behaviourally broken.
. **CI green last 7 days** (C requirement): CI cannot be green — the
  pipelines are YAML-broken as of the merge commit.
. **Deep annotation** (C requirement): the README itself is unreadable.

It also violates the "honest D > dishonest B" clause: self-declared
C + `completion-percentage = 65` coexisting with 20 merge-conflict
files is the *exact* honesty pathology v2.0 was designed to catch.

== Verdict

*Current CRG grade: **E (Pre-alpha)**. Demote two grades from
self-declared C.*

Self-declared: *C*. Audited: *E*. This is the largest honesty delta of
any Bucket-B candidate audited this session (two grades, not one).

Reasoning (one sentence): the repo has committed unresolved
`<<<<<<< HEAD` conflict markers across its README, its MAINTAINERS
file, its TESTING-REPORT, and every RSR workflow touched by the
2026-03-xx merge — so CI cannot parse, the docs cannot render, and
the D-floor "RSR compliance mandatory" clause fails even before the
C-specific clauses (per-directory annotation, CI green, dogfood log)
are considered.

A D grade would require clean workflow YAML and a test-count
reconciliation. A C grade would require those *plus* a populated
`[dogfooding-status]`, per-crate READMEs, and a `docs/` annotation
surface comparable to gossamer's. The gap to C is not one sprint's
worth of work — it is a fix-the-merge sprint *followed by* a
full-completion-percentage-100 sprint.

The E grade is earned, not conservative-by-default, on three positive
signals that keep it above X/F:

1. Real Rust code: 4 non-stub crates totalling ≈2 180 LOC (core 710,
   claude-client 548, lsm 518, esn 404) with constructive content —
   this is not vapourware.
2. Zero dangerous-pattern uses (`believe_me`, `assert_total`,
   `Admitted`, `sorry`, `unsafeCoerce`, `Obj.magic`) in `crates/`.
3. 59 live test attributes including a 14-test proptest integration
   file — tests exist and are non-trivial.

What blocks *immediate* promotion to D (do these first):

. **Resolve the 20 committed merge conflicts.** One commit, careful
  hand-merge of each file, delete all `<<<<<<< HEAD` / `=======` /
  `>>>>>>> ` marker lines. Verify with
  `rg '^<<<<<<< HEAD|^>>>>>>> '` returns zero. Keep the
  PMPL-1.0-or-later SPDX line (drop the `MPL-2.0-or-later` variant per
  Rule 1 of the licence policy).
. **Make `rust-ci.yml` actually run.** Once conflicts are resolved,
  trigger a CI run and record the green SHA in STATE.a2ml.
. **Reconcile STATE.a2ml test counts with reality.** Either write the
  declared suites (smoke/e2e/reflexive/contract/aspect) that are
  missing, or drop their counts. `total-tests` should equal the
  grep-verifiable count plus explicit proptest-case-count if cited.

What blocks promotion to C (do after D):

. Add `[dogfooding-status]` to STATE.a2ml with concrete done-entries:
  what was dogfooded, on which device, when, evidence link.
. Per-crate READMEs for all 8 crates. `neurophone-core/` and `lsm/`
  need full orientation; stubs need a one-paragraph "TODO" placeholder
  so the coverage metric is honest.
. Build out `docs/` with `architecture/`, `decisions/`, `governance/`,
  `practice/` subtrees. Start with ADR-001 explaining the
  LSM+ESN+LLM neurosymbolic architecture.
. Either implement the four stub crates (`bridge`, `llm`, `sensors`,
  `neurophone-android`) or drop them from the workspace until they're
  substantive. `completion-percentage = 65` is more honest than
  declaring C over half-empty crates.

What blocks promotion to B (after C):

. Zero external targets. Need 6+ diverse — different Android OEMs,
  different Android versions, different LLM backends, different
  sensor profiles.
. No external-contributor flow. Dependabot PRs don't count under v2.0.

== Notes on Non-Negotiables Respected by this Audit

- The self-declared grade is tested against v2.0 STRICT criteria as
  written. The "plausibly defensible" classification that queued this
  audit for evidence review stands — the numbers inside STATE.a2ml are
  internally self-consistent. What destroys them is evidence *outside*
  STATE.a2ml: the committed merge conflicts, the missing test files,
  the empty `docs/` tree.
- The four stub crates at 6 LOC are honestly labelled "stub" in
  STATE.a2ml `[crates]`. The audit credits the honesty in labelling
  while still counting the stubs against the "half-complete workspace"
  claim for C.

  (absence of the Android crate, of LLM integration, of sensor wiring
  suggests closer to 35-45%); but since completion-percentage is not a
  v2.0 grade criterion, this audit does not re-score it.
